# serenity-trainer tenets

The non-negotiable principles for the pure-Mojo Serenity/ai-toolkit port. Every
trainer, model, loader, saver, sampler, and dataLoader answers to these. If a
change violates a tenet, the change is wrong — even if it compiles and the loss
goes down.

`docs/PARITY_GATES.md` tells you *what to check*. This document tells you *why*.

---

## 1. The reference is the ONE source of truth — reproduce it 1:1

**For every function, FIRST paste the exact reference code, then translate it —
no deviations, no "improvements", no your-own-cleaner-version.**

- **Trainer / modelSetup / loader / saver / dataLoader / sampler policy**: the
  reference is the Python — **Serenity** (`/home/alex/Serenity`) and, per
  model, **ai-toolkit** (e.g. `extensions_built_in/diffusion_models/ideogram4/`).
  Read the real value off the config/scheduler; never guess a constant.
- **Model forwards / VAE enc-dec / text encoders / sampler numeric cores**: the
  reference is **serenitymojo's working, gated Mojo** (`/home/alex/mojodiffusion`).
  It works — that's why we borrow it.

If you can't find the reference's version of a function, **STOP and ask** — do
not invent it. The failure mode that costs hours: writing your own version,
calling it done, then discovering it diverges.

---

## 2. Measurement beats assertion — the main loop owns every gate

**Never say "works / fixed / parity / matches / done" about behavior without a
tool result in THIS session showing it.** Otherwise it is a HYPOTHESIS.

- "Compiles" is not "works". "Loss goes down" is not "parity". "An image came
  out" is not "a valid image".
- The bar is the **reference's actual numbers**, measured by you on a clean build.
- **Agent self-reports are never the gate.** Re-run the build + the GPU run + the
  numeric comparison yourself. (Worked example: every ideogram4 gate in
  `docs/PARITY_GATES.md` was re-run by the orchestrator on a fresh
  `rm -f serenitymojo.mojopkg` build before it was counted.)

---

## 3. The borrow boundary is fixed

| Layer | Policy |
|---|---|
| `Tensor`, autograd ops (`linear/sdpa/rope/norm/...` + `*_backward`), safetensors io | **IMPORT unchanged** from `serenitymojo.{tensor,ops,io}` |
| Model forwards, VAE enc/dec, text encoders, sampler numeric cores | **COPY/CALL serenitymojo Mojo**, adapt namespace to `serenity_trainer`; ref the borrowed forward against the reference model def |
| trainer / modelSetup / module / loader / saver / sampler / dataLoader | **PORT 1:1 from the Python** (`.py` is the spec) |

A model file holds *architecture*. A trainer holds *loop policy* (schedule,
optimizer, dataset wiring, logging). A primitive workaround that leaks into a
model or trainer file is a bug to migrate back into the primitive.

---

## 4. Train and infer are two paths that MUST agree at B=0

Serenity runs ONE model forward in two contexts: training (autograd, saved
activations) and inference (`no_grad`). In Mojo (hand-chained backward) that is
**two distinct forward functions** — but they compute the same math.

- The sampler MUST use the **inference** forward (no tape, no saved activations,
  no recompute) — reusing the training forward for sampling is not 1:1.
- The training forward (split into embed → LoRA blocks → final to insert LoRA +
  save activations) **MUST reproduce the monolithic inference forward at LoRA
  B=0**, to cosine ≥ 0.999. This is the load-bearing gate.

> **Worked example (the bug this tenet exists for):** the ideogram4 trainer
> "broke" while inference worked. Cause: the split LoRA block sliced AdaLN
> modulation as 2D `[1,Adaln]`, but the real embed produces `[1,1,Adaln]` — the
> monolithic forward handled 3D, the split did not. B=0 velocity diverged from
> the proven predict velocity. Fix: slice on the last axis. The **B=0 gate**
> (split velocity == inference velocity, cos 0.9999373) is what catches this.

---

## 5. Anchor every gate to REAL data, not synthetic

Synthetic random inputs give the **wrong loss scale** — never gate on them. Load
a REAL prepared sample and run it through the REAL path:

- image → VAE encode → latent (ai-toolkit `encode_images`: mean → patchify →
  `(patched - shift)/scale`).
- the **`.json` caption is the prompt** — fed verbatim (raw JSON text) to the
  Qwen3-VL LLM via the chat template, NOT field-parsed.
- noise → predict → velocity/target → loss.

The reference loss is the reference's own number on that sample (ideogram4 giger
sample: `MSE(velocity, target) = 0.961`, in Serenity's 0.2–1.0 range). The
oracle is generated by the reference's OWN runtime (`/home/alex/serenityflow-v2/.venv`),
on GPU, in the production dtype, one model resident at a time.

---

## 6. dtype follows the reference

BF16 storage in/out, F32 only in compute registers — no persistent F32. When the
reference runs a component in a specific dtype (e.g. a VAE built `float32`), the
faithful port runs it in that dtype too; do not lower the oracle to match a
port. The production dtype is truth — measure both dtypes to size any gap, but
gate against production.

A corollary on the loop: cosine is magnitude-blind and accumulation-blind. A
single forward at cos 0.9999 can still drift to 0.96 over an N-step CFG sampling
loop (bf16-kernel divergence amplified by guidance). Gate loops **per-step**, and
check magnitude, not just final cosine.

---

## 7. Chunk, gate, then wire — the integration run IS the wiring test

Build and gate each sub-unit alone (encoder, VAE, predict, train step), THEN
compose. A wrong permute / missing residual / swapped weight tanks the e2e gate
far below the per-unit number — so the end-to-end gate is the test for the
wiring you never unit-tested. One verified deliverable beats three half-built
ones.

---

## 8. Ownership is explicit

The **pipeline** (model/encoder/VAE/sampler/predict/train-step/loader/saver) and
the **UI** (`mojoui` + `src/serenity_trainer/ui/`) are owned separately. Do not
build the same files from two drivers — the shared `serenitymojo.mojopkg` races
on concurrent `mojo` invocations (only ONE process builds at a time). The UI↔
trainer seam is a decoupled protocol (a JSONL command file + stdout progress
lines), not an in-process call — keep both sides to that contract.
