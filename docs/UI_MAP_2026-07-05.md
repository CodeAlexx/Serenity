# Serenity Trainer Mojo UI — Deep Map (build spec for a web replacement)

Date: 2026-07-05
Scope: everything under `src/serenity_trainer/ui/` (34 files, 5904 lines), the
terminal-launch scripts, and the pixi UI build/run tasks. This document is the
completeness-over-brevity reference for a browser + Rust-server replacement of
the native MojoUI trainer screen. All citations are `file:line`.

The UI is a pure-Mojo immediate-mode app built on the `mojoui_floor` library
(`/home/alex/MojoUI`), rendered against an X11/framebuffer backend, that emits
per-model JSON config files and launches campaign-verified `serenitymojo`
trainer binaries in a spawned `gnome-terminal`, then tails their stdout progress
into a live right-rail. There is no in-process training; the UI is a
config-editor + launcher + progress-tailer + media-gallery.

## 2. ARCHITECTURE

### The mojoui_floor frame loop

`main()` (TrainUI.mojo:557) constructs a single `TrainUIAppState`, stores a raw
pointer to it via `store_user_state(sp)` (TrainUI.mojo:559-560), calls
`Backend.init(w, h, "Serenity Trainer")` (TrainUI.mojo:563), then
`Backend.run_blocking(_frame)` (TrainUI.mojo:573). `_frame` is the C-callback the
floor calls once per frame (60 Hz target).

`_frame()` (TrainUI.mojo:534) each frame:
1. Lazily loads the font on first frame (TrainUI.mojo:536-538).
2. `load_preview_textures(media, cfg, runtime)` — gallery scan / throttled rescan
   (TrainUI.mojo:540).
3. `_sync_window` — re-reads window size, reapplies theme + shell density
   (TrainUI.mojo:541, 143-152).
4. `trainer_ui_tick_and_apply(runtime)` — the progress/liveness pump
   (TrainUI.mojo:542).
5. `ctx.begin_frame` → `Backend.frame_begin(bg)` → `_ui(s)` → `ctx.end_frame` →
   `render_context_commands(ctx)` → `Backend.frame_end()` (TrainUI.mojo:543-554).
   Exceptions in `_ui`/`_render` are caught and printed, never fatal.

`_ui()` (TrainUI.mojo:512) lays out the 3-column shell in one row:
`layout_row(row5(nav_w, gap, main_w, gap, status_w), win_h)` (TrainUI.mojo:515),
then `_sidebar`, `_main_panel`, `_status_rail` each inside `begin_column`/
`end_column`, and finally `render_media_lightbox` as an overlay
(TrainUI.mojo:516-527).

### `TrainUIAppState` fields (TrainUI.mojo:78-127)

- `ctx: Context` — the mojoui immediate-mode context.
- `cfg: TrainerUIConfig` — the editable config (loaded from snapshot at init,
  TrainUI.mojo:108).
- `runtime: TrainerUIRuntime` — live run state.
- `font_id: UInt32`, `win_w`/`win_h: Float32`, `metrics: TrainerShellMetrics`.
- `main_scroll_y`, `status_scroll_y: Float32` — scroll offsets.
- `media: TrainerMediaGalleryState`, `captioner: CaptionerScreenState`.
- Ten `TextEditState` fields — the ONLY editable text inputs in the app:
  `run_name_edit`, `workspace_edit`, `cache_edit`, `base_model_edit`,
  `output_dir_edit`, `dataset_path_edit`, `sample_output_edit`,
  `captioner_folder_edit`, `captioner_custom_model_edit`, `captioner_prompt_edit`
  (TrainUI.mojo:91-100). Everything else is a select/slider/drag/toggle bound
  directly to a `cfg` field.

### Section / tab routing

- Section identity is `cfg.section_index: Int32` against the `UI_SECTION_*`
  comptime constants (TrainerConfigModel.mojo:6-17): GENERAL=0, MODEL=1, LORA=2,
  DATASET=3, CONCEPTS=4, TRAINING=5, SAMPLING=6, BACKUP=7, CLOUD=8, RUNS=9,
  LOGS=10, CAPTIONER=11.
- `_sidebar` (TrainUI.mojo:167) draws the "Serenity / Native trainer UI" header,
  a CONFIGURE group of `_nav` rows, a PROJECT group showing `cfg.run_name`. Nav
  order (TrainUI.mojo:187-198): General, Model, LoRA/OFT, Dataset, Captioner,
  Validations, Training, Sampling, Backup, Cloud, Runs, Logs.
- `_nav` (TrainUI.mojo:155) draws one `nav_row`; on click sets
  `cfg.section_index` and resets `main_scroll_y` (TrainUI.mojo:160-163).
- `_main_section` (TrainUI.mojo:299) is the big dispatch: it maps `section_index`
  to the tab renderer. DATASET and SAMPLING also call
  `ensure_visible_*_textures` before rendering and `open_*_preview` after, using
  the click index the tab returns (TrainUI.mojo:306-351). BACKUP/RUNS/LOGS are
  inline functions (`_backup_tab`, `_runs_tab`, `_logs_tab`).
- Section label text comes from `cfg.section_label()` (TrainerConfigModel.mojo:686);
  note it maps CONCEPTS→"Validations" and the label uses "LoRA / OFT".

### Window sizing / theme

- Initial size `_initial_window_size` (TrainUI.mojo:130): 92% × 90% of display,
  floored at `INIT_W=1480 × INIT_H=920` (TrainUI.mojo:74-75).
- Per-frame `_sync_window` (TrainUI.mojo:143) recomputes `TrainerShellMetrics`
  via `trainer_shell_metrics(w, h)`, applies `apply_serenity_trainer_theme(ctx)`
  (from `mojoui.theme.trainer_theme`) and `apply_shell_density(ctx, metrics)`.
  The status rail temporarily clamps font/row height (TrainUI.mojo:403-405) and
  restores them at the end (TrainUI.mojo:508-509).

### Launch environment requirements (the X11 coupling a web UI eliminates)

- **Build:** `pixi run trainer-ui-build` links `TrainUI.mojo` against
  `/home/alex/MojoUI` with `-lmojoui_floor -lm` (pixi.toml:117), `depends-on`
  `build-mojoui` (a `make -C /home/alex/MojoUI/c_floor`, pixi.toml:15) and
  `ideogram4-live-trainer-build` (pixi.toml:19).
- **Run:** `pixi run trainer-ui` = `LD_LIBRARY_PATH=/home/alex/MojoUI
  target/serenity_trainer_ui` (pixi.toml:118). The floor `.so` must be on
  `LD_LIBRARY_PATH`.
- **Display:** the app needs an X11/GNOME session. `Backend.display_size()` and
  `window_size()` drive layout. `Backend.system_metrics()` supplies GPU/CPU stats
  (TrainerRuntimeBridge.mojo:740).
- **Terminal spawn:** the launcher hard-requires a GNOME desktop
  (`gnome-terminal`, D-Bus, `XDG_RUNTIME_DIR`, `DISPLAY`) — see §4. It hardcodes
  `DISPLAY=:1`, `XDG_RUNTIME_DIR=/run/user/1000`, the D-Bus bus address, and a
  CUDA `LD_LIBRARY_PATH` for the spawned trainer
  (scripts/serenity_terminal_launcher.sh env block).

A web UI removes: the X11 backend, the MojoUI floor + textures, the
`gnome-terminal` spawn, and all `DISPLAY`/`XAUTHORITY`/`LD_LIBRARY_PATH` coupling.
The server would spawn trainer processes directly (no terminal window) and stream
their stdout over a socket.

## 4. LAUNCH CONTRACT

### What Start does end-to-end

Start button (TopBar.mojo:62-63) calls `trainer_ui_submit_current(cfg, rt)`
(TrainerRuntimeBridge.mojo:755):
1. Sets `rt.backend_target`/`backend_label` from cfg (756-757).
2. Guards against an already-running run (758-761).
3. `trainer_ui_validate(cfg)` — must return "Ready" or it aborts
   (762-765). Validation checks: base model present, dataset path present,
   lora_rank≥1, lr>0, and LTX2-only LyCORIS block (TrainerConfigModel.mojo:1033).
4. Capability-table honesty: `trainer_ui_ignored_lever_summary(cfg)` — logs a
   WARNING naming every non-default widget the runner does not consume, but does

5. `_save_config_snapshot(cfg)` → writes `target/serenity_trainer_ui_config.json`

6. Bumps run_id, resets live stats, records progress-file byte offset baseline,

7. Appends a `start` event to the command file
   `target/serenity_trainer_commands.jsonl` (814, `_append_command_event`).
8. `trainer_ui_launch_live_runner(cfg, rt)` (816) — the actual spawn.
9. On success sets `rt.has_running=True` and appends a run to `rt.runs` (820-821).

### serenity_terminal_run.sh's role

The launcher `scripts/serenity_terminal_launcher.sh` (invoked above) does two
things: (a) **heredoc-writes** `target/serenity_terminal_run.sh` fresh each launch
(so that file is a generated artifact, not source), then (b) `env -i` scrubs the
environment and re-exports a fixed GNOME + CUDA env (HOME, USER, DISPLAY=:1,
XDG_RUNTIME_DIR=/run/user/1000, DBUS_SESSION_BUS_ADDRESS, PATH with CUDA,
LD_LIBRARY_PATH with CUDA libs) and launches `/usr/bin/gnome-terminal
--title="Serenity <label> Trainer" -- bash <terminal_script> <root> <pidfile>
<logfile> <label> <runner> <args>`.

`target/serenity_terminal_run.sh` itself (the generated wrapper): cd's to root,
writes its OWN pid to `<pidfile>.term` (the terminal-wrapper pid), prints a
banner, runs `"$runner" "$@" > >(tee "$logfile") 2>&1 &`, writes the child
runner pid to `<pidfile>`, installs a `trap stop_child TERM INT HUP` that
TERMs the child + cleans pidfiles + exits 143, `wait`s on the child, and on
nonzero exit shows a 30s "closing" prompt (success/stopped closes immediately).

Because `gnome-terminal` returns immediately (spawn goes over D-Bus to
gnome-terminal-server), the launcher polls for up to 20s for `<pidfile>` or
`<logfile>` to appear, and returns nonzero if the runner never started ("training
started but no terminal opened" guard).

**pid files** (per backend, `_pid_path`/`_term_pid_path`, 264/484):
`target/serenity_<backend>_live_trainer.pid` (runner pid) and `.pid.term`
(terminal-wrapper pid). ideogram4 uses `target/serenity_ideogram4_live_trainer.pid`.

**progress log** (`_log_path`, 270): config/hidream/krea2 runners tee stdout
straight into `target/serenity_trainer_progress.log` (the polled progress file);
ideogram4 tees into `/tmp/serenity_ideogram4_live_trainer.log`.

There is also an older `target/serenity_ideogram4_terminal_run.sh` and
`scripts/serenity_ideogram4_terminal_launcher.sh` — a legacy ideogram4-specific
variant (keeps a "Press Enter to close" hold). The current code path uses the
generic launcher.

### Process lifecycle (stop / kill)

- **Stop** (TopBar.mojo:59) → `trainer_ui_cancel` (TrainerRuntimeBridge.mojo:848):
  if an owned runner is active, `_stop_live_runner` (494) TERMs the terminal
  WRAPPER pid first (its trap tears down the runner AND closes the window), falls
  back to TERMing the runner pid, and removes the pidfiles only when a kill
  landed. Sets rt to Stopped, appends a `stop` command event.
- **Liveness guards**: `_owned_live_runner_active` (468) requires the pidfile pid
  to be alive AND its `/proc/<pid>/cmdline` to actually be this runner (defends
  against recycled pids from stale pidfiles). `_any_live_runner_active` (463)
  `pgrep`s for any serenity trainer.
- **Auto-detect exit**: `trainer_ui_tick_and_apply` (1212), ~once/sec after a 3s
  grace, checks `_owned_live_runner_active`; if the process vanished it calls
  `_record_live_runner_exit` (516) — re-reads the log for a final progress line,
  removes pidfiles, sets "Trainer process exited".

### Resume argv (the MJ-1077 trap)

**UNVERIFIED / GAP:** There is NO resume-with-state argv in the UI code. The
ideogram4 argv has an "argv 16 = optional resume LoRA path" slot but the UI
always passes `-` (skip sentinel) for it (TrainerRuntimeBridge.mojo:399 emits a
literal `-`). `continue_last_backup` is a General-tab toggle that is snapshot-only
(never reaches any argv). So the known MJ-1077 trap — full resume needs the
`.state` path — is **not implemented**: the UI cannot resume a run from optimizer
state; a relaunch starts fresh. A web UI must add this if resume is required.

## 5. PROGRESS + STATUS

### Progress log tailing / parsing

The pump is `trainer_ui_tick_and_apply` (TrainerRuntimeBridge.mojo:1212), called
every frame from `_frame`:
- Every 60 frames refreshes system metrics (1214-1216).
- If running and not paused, every 6 frames (~100 ms) calls
  `trainer_ui_poll_progress_file` (1224). That reads the WHOLE progress file,
  applies only bytes past the last offset (`rt.progress_file_bytes`), splits on
  `\n`, and feeds each line to two parsers (`_apply_progress_text_from`, 1167).
- The file re-read is intentionally throttled off the 60 Hz render thread (the
  file is tee'd full trainer stdout, grows monotonically).

Two accepted line shapes:
1. **Serenity callback** (`trainer_ui_apply_serenity_callback_line`, 1004):
   `[Serenity-callback] progress epoch 1/10 | step 44/120 | global_step 164 |
   loss 0.5909 | smooth_loss 0.6123 | grad_norm 0.1527 | lr 0.000100 | status
   Training`. Extracted fields: epoch/total_epoch, step/total_step, global_step,
   loss, smooth_loss, grad_norm, lr, status.
2. **Klein/tqdm progress** (`trainer_ui_apply_progress_line`, 1067):
   `[Klein-lora] step 1613/2000 | epoch 14/17 | loss 0.5909 | grad_norm 0.1527 |
   2.1s/step | elapsed 0:55:37 | ETA 0:13:20`. Extracted: step/total_steps,
   epoch/total_epochs, loss, grad_norm, optional smooth_loss (falls back to
   UI-side EMA), s/step speed, ETA seconds. Global_step is derived as
   `(epoch-1)*total_steps + step`.

Parsing is done entirely on RAW BYTES (`_find_token`, `_find_char`, `_token_end`,
`_read_int_between`, `_read_float_between`, `_parse_seconds`) because trainer
stdout can contain multi-byte UTF-8 and String codepoint indexing at an arbitrary
byte offset asserts (TrainerRuntimeBridge.mojo:905-908 documents this). Both
parsers parse into locals FIRST and commit only after the whole line validates —
this is the fix for inline-sampler lines corrupting the step counter (bug 1,
audit 2026-07-03).

Completion is `_live_progress_complete` (723): total reached, or step≥total and
epoch≥total_epochs. ETA is recomputed by `_update_eta_like_serenity` (637) from
wall time / steps-done × remaining-steps (1-based epoch clamp).

### The status rail (`_status_rail`, TrainUI.mojo:398)

Fields shown: RUNNING/PAUSED/IDLE pill, a progress bar
(`trainer_ui_progress_fraction`), then Step, Epoch, Loss, Smooth, Grad, LR, Speed
(s/step), ETA (`trainer_ui_eta_label`), Status, Command, Source
(callbacks/progress-file/waiting). Artifacts block: Samples count, Checkpoints
count, Backend. Hardware block: GPU Model, Driver, GPU util%+temp, VRAM used/total,
CPU Model, CPU util%, RAM used/total.

GPU/CPU stats source: `trainer_ui_refresh_system_metrics`
(TrainerRuntimeBridge.mojo:739) calls `Backend.system_metrics()` (a MojoUI
backend API) → gpu_name, gpu_driver, gpu_util_percent, gpu_temperature_c,
gpu_memory_used/total_mb, cpu_name, cpu_util_percent, ram_used/total_mb. The web
UI must replace this with a server-side probe (nvidia-smi / /proc).

### Run history

`rt.runs: List[String]` — appended in `trainer_ui_submit_current`
(TrainerRuntimeBridge.mojo:821) as `"#<id> <run_name> - <backend> target"`.

## 6. MEDIA GALLERY

### Sample / dataset discovery

`load_preview_textures` (TrainerMediaGallery.mojo:465) is called every frame from
`_frame`. It keys on `dataset_path + "\n" + sample_output_dir`; on first sight it
`_load_scanned_gallery` (273) which scans both dirs via
`Backend.scan_media_files(dir, recursive=True, cap=4096)` (217/245). Dataset
tiles preload the first 48 textures, sample tiles the first 32; the rest load
lazily on scroll. Titles are "Dataset N"/"Sample N"; subtitles use `_basename` of
the file path (image/video prefix for samples).

While training, a throttled (~3 s / 180 frames) count-only rescan
(`_refresh_sample_gallery`, 430) rebuilds the sample gallery iff the file count
changed — this is how new inline samples appear during a run. It is skipped while
a lightbox preview is open (so the active texture is not destroyed).

### Texture load pipeline

- `_load_media_texture` (170) → `Backend.load_video_thumbnail_info` for videos
  else `Backend.load_texture_file_info(path, max_w, max_h)`.
- Virtual scrolling: `ensure_visible_dataset_textures`/`_sample_` (354/382) plus
  `_ensure_media_range` (309) compute the visible tile window from scroll_y and
  gallery geometry (`gallery_columns`, `gallery_card_height`, `gallery_visible_rows`,
  279-306), load textures in `[start,end)`, and destroy textures outside a
  ±96-tile keep band to bound VRAM. The active lightbox texture is never destroyed.
- Grid render uses the shared MojoUI `media_virtual_grid` widget (DatasetTab.mojo:93,
  SamplingTab.mojo:81, CaptionerTab.mojo:392). Column count = content_w//360
  clamped [2,8]; card ~4:3.

### Preview interaction (lightbox)

Clicking a tile returns its index from the grid widget; `_main_section` calls
`open_dataset_preview`/`open_sample_preview` (TrainerMediaGallery.mojo:535/560)
which lazily upgrade the tile to a 960² texture and set the `active_preview_*`
state. `render_media_lightbox` (585) draws the overlay via the MojoUI
`media_lightbox` widget; actions: `MEDIA_LIGHTBOX_PLAY` → `Backend.open_video_file`
(external player) for videos, `MEDIA_LIGHTBOX_CLOSE` → clears active preview.

### The UTF-8 `_basename` fix (commit 2c81bbb)

`_basename` (TrainerMediaGallery.mojo:156) scans RAW BYTES for `/` (0x2F) and
slices at `last+1`, which is always a codepoint boundary because `/` is
single-byte ASCII. The comment explicitly notes that the naive
`String(path[byte=i])` per-char form asserts when byte i lands mid-codepoint —
i.e. it crashed on any non-ASCII path char. This is the gallery's version of the
crash class that still lives in CaptionerTab (see §8).

### Video handling

Video thumbnails are detected two ways: `Backend.is_video_file(path)` and a
thumbnail-path convention `<name>.<ext>-...` (`_video_thumbnail_source_end`, 87,
recognizing .mp4/.mov/.m4v/.mkv/.avi/.webm). `_video_source_path` (113) strips the
thumbnail suffix back to the source video for playback. Sample/dataset subtitles
tag "video" vs "image".

## 8. KNOWN DEBT

1. **Cross-preset contamination (run_name/workspace/prefix leak).** Only the
   Krea-2 preset sets run_name / workspace_dir / sample_output_dir /
   save_filename_prefix (TrainerConfigModel.mojo:995-1018). No other preset
   resets them. Selecting Krea-2 then switching to Klein (or any other model)
   leaves `run_name="eri2_krea2_lora"`, `workspace_dir=SERENITY_KREA2_WORKSPACE`,
   `save_filename_prefix="eri2_krea2"` in place — the Klein run then displays and
   snapshots Krea-2's naming. The web UI should reset these per preset.

2. **Klein stale `alina` naming + UI/trainer output-dir mismatch.** The klein
   launch mkdir hardcodes `/home/alex/mojodiffusion/output/alina_train`
   (TrainerRuntimeBridge.mojo:429) and the comment states `train_klein_real`
   saves LoRAs/samples into its comptime `LORA_DIR`/`SAMPLE_DIR` (alina_train).
   Meanwhile the Klein preset never sets workspace_dir, so the UI shows the struct
   default `/home/alex/trainings/boxjana_klein9b_lora_v1` (TrainerConfigModel.mojo:538)
   — the displayed workspace does not match where klein actually writes. The klein
   cache constant also carries a "was alina_klein9b" provenance note
   (TrainerConfigModel.mojo:20).

3. **Empty caps-cache path crash (`cap_cache.header open failed`).**
   **UNVERIFIED from UI code:** this is a downstream `train_klein_real` behavior —
   when the klein cache dir is empty/missing the trainer fails opening the caption
   cache header. The UI does not preflight the cache; it will happily launch a
   klein run against a missing `SERENITY_BOXJANA_KLEIN_CACHE`. A web UI should
   validate cache existence before launch. Related: `SERENITY_ZIMAGE_CACHE` is
   annotated MISSING (TrainerConfigModel.mojo:65) and `SERENITY_L2P_CACHE` has no
   prepared cache (line 67-69) — those presets launch straight into a fail-loud
   trainer preflight.

4. **Remaining byte-indexed String slicing crash sites.** Grep of `ui/` for the
   single-byte `String(x[byte=i])` form (asserts on multi-byte UTF-8) finds THREE
   live sites, all in **CaptionerTab.mojo**: `_basename` line 68, `_sidecar_path`
   line 79, `_json_escape` line 111. These crash the captioner on any non-ASCII
   path or caption byte — the same class the media gallery already fixed with a
   raw-byte scan (TrainerMediaGallery.mojo:156-167) and the progress bridge avoids
   (TrainerRuntimeBridge.mojo:905-908). (Range slices `[byte=a:b]` at known
   boundaries are safe and used widely; the dangerous form is the single-index
   `[byte=i]` codepoint access.) The web UI, being string-native, eliminates this
   class entirely.

5. **[not wired] widgets that must not silently lie.** Several widgets are
   snapshot-only — no runner consumes them: Aspect Buckets (DatasetTab.mojo:53),
   the entire MASKED TRAINING panel (TrainingTab.mojo:158-165), Train Method
   Fine-Tune/Embedding (ModelTab.mojo:58), Pause/Resume/Sample-now (TopBar.mojo:82-90,
   the command file is written but no trainer reads it). The capability table
   (`trainer_ui_supported_lever_keys` / `_active_lever_keys` /
   `_ignored_lever_summary`, TrainerConfigModel.mojo:1080-1156) drives a
   "<backend> ignores: …" WARNING in the top bar and submit log so non-consumed
   non-default widgets are named loudly rather than silently dropped. Per-backend
   consumed levers: klein/zimage/hidream/ideogram4 = optimizer, warmup, loss_fn,
   min_snr_gamma_flow, ema, caption_dropout, network_algorithm; krea2 = those
   minus ema/caption_dropout; chroma/ernie/anima/sdxl/l2p/qwen = network_algorithm
   only.

6. **Hardcoded values that violate the config-driven rule.** Beyond the mkdir dirs
   in §2: several tab panels render fixed placeholder strings instead of live
   config (e.g. Dataset "Buckets: 704x1024, 1024x1024, 1024x704"
   DatasetTab.mojo:63; LoRA "Targets: qkv, proj, mlp, modulation" LoraTab.mojo:57;
   almost every aux/window surface in §1). The aux windows in particular are
   Rust-snapshot parity stubs full of hardcoded field text and are not wired to
   the running app (imported only by the compile test
   `tests/ui/test_aux_screens_compile.mojo`).

7. **sd35 blocked route.** model_type STABLE_DIFFUSION_35 (index 3) has no
   architecture entry; `_arch_index_for_model_type` returns -1
   (TrainerConfigModel.mojo:786), so `trainer_ui_apply_model_preset` routes to
   `backend_target="sd35"`, `model_arch="sd35"` and returns early
   (TrainerConfigModel.mojo:826-829). At launch, sd35 matches no wired-runner
   guard, so `trainer_ui_launch_live_runner` logs "no live trainer runner wired
   for backend: sd35" and status "No trainer wired for sd35" and returns False
   (TrainerRuntimeBridge.mojo:551-556) — fail-loud, no silent fallback to the
   previous model. (Consistent with the frozen-sd3/sd35 mandate.)

8. **Resume not implemented** — see §4 (MJ-1077): the argv resume slot is always
   `-`; `continue_last_backup` never reaches an argv.

## 9. WEB UI MAPPING (bridge, not design)

- Config model (`TrainerUIConfig`) → `GET/PUT /api/config` (JSON == the
  `trainer_ui_config_json_snapshot` schema `serenity.trainer_ui.v1`).
- Preset table (`trainer_ui_apply_model_preset`) → `GET /api/presets` and
  `POST /api/config/preset {architecture_index}` (server applies the preset,
  returns the merged config).
- Architecture/model-type selectors + sync maps → part of `/api/presets` metadata
  (the 13-row table + the arch↔model_type mapping).
- Runner JSON emission (`trainer_ui_runner_train_config_json`) → server-side on
  launch; not a client concern.
- Start → `POST /api/runs {config}` (server validates, writes runner config,
  spawns the trainer process directly — no terminal — returns run_id).
- Stop → `POST /api/runs/{id}/stop`; Pause/Resume/Sample/Save →
  `POST /api/runs/{id}/{action}` (command-file semantics; note most are [not wired]).
- Progress + status rail → `WS /api/runs/{id}/events` streaming the parsed
  step/loss/grad/lr/eta + status; system metrics via the same socket or
  `GET /api/system/metrics`.
- Validation + capability warning → `GET /api/config/validate` returning the
  "Ready"/error + the "<backend> ignores: …" summary.
- Runs list / logs → `GET /api/runs`, `GET /api/runs/{id}/logs`.
- Dataset gallery → `GET /api/dataset/media?path=…` + static file serving of
  images/video thumbnails; lightbox = client-side.
- Sample gallery → `GET /api/runs/{id}/samples` (or `?dir=sample_output_dir`) +
  static serving; poll/stream for new samples during a run.
- Captioner → `GET /api/captioner/models`, `POST /api/captioner/scan {folder}`,
  `POST /api/captioner/run {settings}` (replaces the JSONL command file); sidecar
  read/write via `GET/PUT /api/captioner/caption?path=…`.
- Config snapshot persistence → server-owned; the fixed
  `target/serenity_trainer_ui_config.json` becomes a per-user/server config store.
- Everything under §1 "aux/window surfaces" → future tabs; not needed for parity.
