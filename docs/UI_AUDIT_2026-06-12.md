## P0.1 Terminal does not close when trainer stops/finishes — MEASURED, FIXED

Mechanism (reproduced live with a sleep-300 dummy runner):
- `scripts/serenity_terminal_launcher.sh` heredoc wrapper backgrounds the runner; **runner shares the wrapper bash's process group** (measured: wrapper pid 3816444 pgid 3816444, runner 3816447 pgid 3816444).
- UI stop path (`TrainerRuntimeBridge.mojo _stop_live_runner`) did `kill -TERM -- -$runner_pid` → **"No such process" (rc=1, measured)** because the runner pid is not a pgid; fallback killed only the runner.
- Wrapper survived (measured: `WRAPPER STILL ALIVE` 2 s after stop) and fell into the unconditional `read -r -p "Press Enter to close terminal..."` (old launcher line 61) → window open forever. Same hold on **natural completion**.
- Fix direction verified before editing: `kill -TERM <wrapper>` → trap fires → runner killed → wrapper exits → window closes.

Fix (launcher + bridge, one protocol):
- `scripts/serenity_terminal_launcher.sh`: wrapper now writes its own pid to `"$pidfile.term"`; trap extended to `TERM INT HUP` (HUP = window-X/logout previously bypassed cleanup); trap kills runner, removes both pidfiles, exits immediately. Natural exit: status 0 → close immediately; status != 0 → `read -t 30` (error visible 30 s, then auto-close; full output is in the logfile anyway).
- `TrainerRuntimeBridge.mojo`: new `_term_pid_path()`; `_stop_live_runner` TERMs the wrapper first (closes window via trap), falls back to the runner pid, removes both files; `_record_live_runner_exit` and the launch command's `rm -f` clean both files.

Verified (live, end-to-end):
```
launcher rc=0 latency=.14s
runner pid: 3817453 / wrapper pid: 3817450
stop ok=0
PASS wrapper exited (terminal window closed)
PASS runner dead
PASS natural completion auto-closed terminal   (pidfiles also cleaned)
```

## P0.2 Training start sometimes opens no terminal — mechanisms pinned, FIXED

Could not reproduce the intermittent symptom on demand; three concrete launch-path defects were pinned and fixed:

1. **Fire-and-forget spawn (MEASURED code path):** `gnome-terminal` is a D-Bus client to `gnome-terminal-server` (server pid 3616580 running, DISPLAY=:1, x11); it returns rc 0 the moment the RPC is accepted — **before** any window/wrapper/runner exists. Old launcher exited right there, so a server-side spawn failure produced exactly "training submitted, no terminal, no error". FIX: launcher now polls up to 20 s for the runner pidfile (or the tee-created logfile, covering instant-exit runners) and exits 1 if neither appears → bridge surfaces "Live trainer launch failed" instead of silence. Measured launch latency idle: 0.14 s.
2. **Stale-pidfile false "already running" (HYPOTHESIS for the recycled-pid trigger, MEASURED guard defect):** `_owned_live_runner_active` was `kill -0 $pid` only. SIGKILL/old-HUP paths leave the pidfile behind; any recycled live pid then blocks every launch with "Trainer already running" → no terminal. FIX: pid must also match the runner path in `/proc/$pid/cmdline` (`grep -aq`). Verified: non-runner live pid now rejected (`PASS: cmdline check rejects non-runner pid`).
3. **HUP leak:** window closed via X sent HUP (untrapped) → wrapper died without removing pidfiles, feeding (2). FIX: HUP now trapped (see P0.1).

Residual HYPOTHESIS (not fixed, low priority now that launch is synchronous): `trainer_ui_tick_and_apply`'s 3 s grace (`TrainerRuntimeBridge.mojo:~1090`) could fire before a >3 s terminal spawn on a loaded desktop; with the launcher now blocking until the pidfile exists, the pidfile is guaranteed present before the first tick, so the race is closed in practice.

Note: closing the **UI** never touches a running trainer/terminal (no teardown hook in `TrainUI.mojo main()`); training survives UI exit. Looks intentional (long runs), flagged for Alex to confirm.

## P2 Performance — two per-frame costs FIXED, rest report-only

MojoUI loop = sokol `sapp_run` (c_floor/mojoui_platform.c:235): the frame callback runs every vsync; anything inside blocks the render thread.

FIXED (TrainerRuntimeBridge.mojo `trainer_ui_tick_and_apply`):
- `trainer_ui_poll_progress_file` re-read the **entire** progress file every frame (`_read_file_text` = full `f.read()`); for config runners the file is the tee'd full trainer stdout, growing monotonically → O(bytes) per frame at 60 Hz, O(n²) over a run. Now polled every 6 frames (~100 ms).
- `_owned_live_runner_active` forked a shell (`system()`: cat+kill) **every frame** while running without fresh progress. Now 1 Hz (`frame_counter == 30`).

Report-only suspects (HYPOTHESIS unless noted):
- `_file_byte_length` (bridge :~1016) reads the whole file just to measure it (used once per submit — fine, but wrong tool; `seek` unused anywhere in the codebase).
- Typing in the Dataset-path/Sample-dir text fields changes `source_key` per keystroke → `load_preview_textures` destroys + rescans + reloads up to 48+32 textures **per keystroke** (MEASURED from code; impact unmeasured). Debounce wanted.
- `_logs_tab`/`_status_rail` render is bounded (last 10 / fixed rows) — fine. Logs ring buffer pops front of a 512-entry List per append (O(n) shift) — negligible.
- Immediate-mode full redraw per frame is by design (no damage tracking); not a defect.

## P4 Widget inventory (training-relevant)

Text edits (10 `TextEditState`s in TrainUI.mojo :90-99 — run_name, workspace, cache, base_model, output_dir, dataset_path, sample_output, captioner×3):
- Engine = real stb_textedit port (caret/selection/undo/word-nav/Home/End) — solid.
- **No system clipboard**: `ctx.clipboard` is in-app only; zero clipboard FFI in MojoUI (grep over render/ffi.mojo + c_floor = no hits). **Ctrl+V cannot paste a path copied from another app** — major usability gap for a path-heavy trainer UI (report-only; needs new C-floor FFI).
- Byte-based cursor: multi-byte UTF-8 can be split (documented in widget header) — ASCII paths fine.
- Commit semantics: live mutation per keystroke, no enter/blur commit, no validation; dataset-path keystrokes trigger gallery rescans (see P2).

Numeric fields: ALL numbers are `drag_value` scrubs or sliders — **no typed numeric entry anywhere, no min/max clamp on drags** (drag_value.mojo: "No re-snap, no momentum, no clamping"). Consequences: Warmup/LR/Seed can be dragged negative; learning_rate at 1e-5/px is near-impossible to set precisely; seed is a Float32. select_string_row/toggle_row are sound. `field_row` is display-only — several look editable but aren't (Frames, Sequence Len, Filter, Live Steps).

## P5 Hardening list (report-only)

- **Pause/Resume/Sample-Now/Save-Now are no-ops at the trainer**: they append to `target/serenity_trainer_commands.jsonl`, but **no trainer reads that file** (grep across serenitymojo/training + serenity-trainer/trainer = zero consumers). "Paused" is pure UI state; the trainer keeps running. (zimage has its own sample_requests dir protocol, written by the trainer itself, not by the UI.)
- No config persistence: UI writes `target/serenity_trainer_ui_config.json` snapshots but never loads anything at startup — every launch is hardcoded defaults.
- Runner exit status is never surfaced: `_record_live_runner_exit` ingests the log tail but swallows read errors (`except: pass`) and reports no exit code; the wrapper knows `$status` but writes it nowhere.
- `trainer_ui_poll_progress_file` swallows all I/O errors (`except: return False`) — a permission/disk error looks like "no progress".
- `_shell_quote` does not escape embedded single quotes (acknowledged comment); a path with `'` breaks the launch command.
- Zombies: none — wrapper reaps its child; the runner is parented to gnome-terminal-server, not the UI.
